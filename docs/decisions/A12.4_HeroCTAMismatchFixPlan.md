## Root Cause Analysis

### Why CTA is Not Rendered
1. **Original Zivan Template:** The Hero component in the Zivan template does NOT include a CTA button
2. **1:1 Lock:** Zivan frontend must remain pixel-perfect to original template
3. **Design Intent:** Hero uses VideoModal as primary CTA, not a text button

### Why CTA Fields Exist in Admin
1. **Future-proofing:** Schema includes CTA fields for potential future use
2. **Standard Pattern:** Hero sections commonly have CTAs in other templates
3. **No Harm:** Fields exist but are simply not consumed

## Fix Approach Options

### Approach A: Hide/Remove CTA Fields in Admin (RECOMMENDED)

**Description:**  
Temporarily hide or remove CTA fields from the Hero admin form until public UI supports rendering them.

| Aspect | Assessment |
|--------|------------|
| **Action** | Comment out or remove CTA inputs from HeroForm.tsx |
| **Impact** | Admin form matches public capability |
| **Data** | Existing CTA data remains in DB (no loss) |
| **Reversibility** | Easy — uncomment when public supports CTA |
| **Risk** | Low — no public UI changes |

**Rationale:**  
- Prevents user confusion (editing fields that have no effect)
- Maintains Zivan 1:1 lock on public UI
- Clean Admin UX — only show what works

Add CTA button rendering to the public Hero component.

| Aspect | Assessment |

**Rationale:**

## Recommendation

### ✅ Approach A: Hide CTA Fields in Admin

**Reason:**  
The Zivan 1:1 lock is a non-negotiable constraint. Since the original Zivan Hero does not include a CTA button, we cannot add one to the public UI. Therefore, the Admin should not expose CTA fields that have no effect.

**Status:** This is the recommended approach, but **NO implementation in this phase**.
